# 关于开发


## v3.* 更新重新回归先行与稳定两个通道
+ 主战场将移动到mirror`内测版`通道,标志为`带有beta`的版号(命名规则:Beta版号比正式版+1补丁版本)。
+ `正式版`会更稳定，但也将比现在更大的落后于最新`内测版`。（填补时间约等于‘bug被发现时间’+‘修复’+‘修补补丁被验证时间’乘以轮数 ）

### 资源更新通道切换可在`设置`最底找到选项

> 早先`V2.*`中维持了较高的更新速度，推动大跃进同时也发现了问题——迭代速度太快。单主分支集合新功能、Debug、重构，测试忙不过来，往往之前反馈回来后面另一个功能的初版已经提交，于是紧急修复发版里附带了其他功能的初版代码。异步开发，同时多个功能既需要测试又需要隔离。总之弄得有些狼狈。  故停下来思索，得出了这套实验性方案，虽然操作起来复杂了亿点点，但在满足需求同时将选择权给到用户——是否有精力、愿意协助开发，还是坚持虽然功能落后但安稳地留在一个问题少的版本？  个人希望大家能参与`内测版`甚至[回报问题](https://github.com/sunyink/MFABD2/issues?q=is%3Aissue) ，真正的头版代码都会由我和测试小组先跑ci开发版，内测版没有那么那么不靠谱( • ̀ω•́ )✧，我们只是人不够维持不了太大范围高覆盖测试率，而且就算有虫子，也有信心快速响应干掉他（确信~）。

## v3.* 更新机制讲解，将尝试类似DNA双链演进的方法继续更新：
+ 区分更新更新通道`正式版`、`内测版`分别对应`Main分支`、`Develop分支`。
+ 一般的，这两个分支均不允许直接提交，而是**以功能分支的合并或PR推动2个主分支向前更新！**。
  - 禁止变基、压缩合并。带上`--no-ff`参数!
+ 每一个需求新建独立分支提交(功能分支)，完成后合并入`内测版`。后续相关bug也全部在独立分支进行。最后相当稳定再独立PR并入`正式版`。
+ 功能分支开发告一段落，合并入`Develop分支`上传，一般要求在内测最新版本上测试，防止事后功能分支之间互相冲突。
+ 测试覆盖率、再次检视均没再发现问题，以功能分支为单位PR到正式版`Main分支`。PR成功后远程功能分支默认删除。
+ 从此抛弃分支内时间线，以功能分支为单位去推进版本。
